⏺ Done. Here's what I changed and how to ship it.

  What was wrong

  backend/services/vllm.py opened a stream=True chat completion to vLLM and forwarded aiter_lines(). If vLLM replied 4xx (or anything non-streamy), the lines were
  forwarded silently and the UI just hung blank. The blog's working pattern is a plain non-streaming POST.
  
  What's new

  backend/services/vllm.py now:

  - Uses client.post(..., json=payload) with model / messages / max_tokens — identical request shape to the blog
  - No stream: true — exactly the blog's payload
  - Caps each context chunk to 500 chars and skips chunks that start with [ (the "vector leakage" guard from the blog)
  - On non-200, emits an SSE event: error with the status + body so the frontend can show the actual problem instead of hanging
  - Re-emits the answer as one OpenAI-style SSE delta + [DONE], so RagQuery.tsx parses it unchanged (no frontend edit needed)
  
  Ship it on Windows
  
  Pull, rebuild the in-cluster image, and roll the deploy. From the Windows WSL shell where the agent runs:
  
  /bash cd ~/cso-operator-app && git pull
  
  /bash cd ~/cso-operator-app && eval $(minikube docker-env) && docker build -t cso-operator-app:latest .
  
  /bash kubectl rollout restart deploy/cso-operator-app && kubectl rollout status deploy/cso-operator-app
  
  Then submit a prompt in the UI. If anything still fails, the error is now visible:
  
  /bash kubectl logs deploy/cso-operator-app --tail=120
